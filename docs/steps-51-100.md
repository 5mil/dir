## Phase 6 — Advanced Features (Steps 51–60)

### Step 51 — Implement entity suggestions API: autocomplete on title slug with fuzzy matching

**Why:** Add fast autocomplete for entity title/slugs to improve authoring and search UX.

**How:** Use Postgres trigram index (`pg_trgm` extension). Query: `SELECT slug, title FROM dir_entities WHERE title % $1 ORDER BY similarity(title, $1) DESC LIMIT 10`. Expose via `GET /api/entities/suggest?q=...`. Cache hot queries in Redis with 60s TTL.

---

### Step 52 — Build 'related entities' widget: query graph for entities with shared tags or relations

**Why:** Surface contextually related knowledge nodes to increase content discoverability.

**How:** Graph query using `dir_relations` (shared relation_type) and `dir_entity_tags` (shared tag_id). Score by relation weight + credibility_score. Limit to top 5. Cache in Redis keyed by entity slug.

---

### Step 53 — Implement multi-language support: add language field to entities, translations table

**Why:** Allow entities to exist in multiple languages to serve a global audience.

**How:** Add `dir_entity_translations` table: `(id, entity_id, language VARCHAR(8), title, summary, body, status, created_by, created_at)`. Add `language` field to `dir_entities` as primary language. Add locale-aware routing: `/en/entity/:slug`, `/es/entity/:slug`.

---

### Step 54 — Create translation workflow: draft translation, approve by board, publish to entity

**Why:** Govern translated content quality through the same governance modes as primary content.

**How:** Reuse revision + governance ACL from `plugin-dir-governance`. Translation edits create `dir_revisions` with `edit_type: translation`. Board approval required in board mode before `status = published`.

---

### Step 55 — Add media support: upload images, PDFs, store in S3 or local storage, link to entities

**Why:** Enrich entity pages with diagrams, portraits, source PDFs, and archival images.

**How:** Create `dir_media` table: `(id, entity_id, media_type, filename, url, storage_backend, size_bytes, alt_text, created_by, created_at)`. Use MinIO for self-hosted or AWS S3 for cloud. Serve via signed URLs. Enforce 50MB max upload per file.

---

### Step 56 — Implement rich text editor: Markdown support, inline citations, fact highlighting

**Why:** Authors need a structured, citation-aware editor to build high-quality entries.

**How:** Integrate TipTap (ProseMirror-based) with custom Citation extension. Store content as structured JSON (TipTap doc model). Sanitize with DOMPurify on save. Render as HTML on page load.

---

### Step 57 — Add 'see also' section: manually curated cross-references between entities

**Why:** Allow authors to surface non-graph, editorial cross-links for reader navigation.

**How:** Add `dir_see_also` junction table: `(entity_id, related_entity_id, note, display_order)`. Render as inline card list at bottom of entity detail page.

---

### Step 58 — Implement entity version comparison: diff view between any two revisions side-by-side

**Why:** Let editors and reviewers compare any two historical snapshots of an entity.

**How:** Use `dir_revisions.snapshot` (JSONB). Compute deep-diff on server using `json-diff` library. Render side-by-side as colored line diff. Expose at `GET /api/entities/:id/diff?from=3&to=7`.

---

### Step 59 — Add 'did you mean?' suggestions: spelling correction for search queries

**Why:** Improve search recall by correcting common misspellings in user queries.

**How:** Use Postgres `pg_trgm` to find closest-matching entity titles. Alternatively, integrate Levenshtein distance function. Return `did_you_mean` field in search API response alongside results.

---

### Step 60 — Create API documentation: Swagger/OpenAPI spec for all public endpoints

**Why:** Enable third-party integrations and developer onboarding with self-documenting API.

**How:** Generate OpenAPI 3.0 spec. Publish Swagger UI at `/api/docs`. Include all collection endpoints, custom actions (publish, revise, credibility, rollback), authentication schemes, and example payloads.

## Phase 7 — Scaling & Performance (Steps 61–70)

### Step 61 — Add Redis caching layer: cache entity detail pages, search results, hot queries

**Why:** Reduce database load and serve repeat visitors at microsecond latency.

**How:** Deploy Redis 7 as a sidecar container. Use `ioredis` client. Cache entity pages with key `entity:slug:{slug}`, TTL 5 min. Cache search results with key `search:{hash(query+filters)}`, TTL 2 min.

---

### Step 62 — Implement cache invalidation strategy: TTL, on-write eviction, per-user session cache

**Why:** Ensure users never see stale content after an entity is updated.

**How:** On `dir_entities` update, run `MULTI/EXEC` to delete `entity:slug:*` matching slug. Use pub/sub channel `dir:invalidate` for distributed invalidation across multiple app instances.

---

### Step 63 — Add database connection pooling: configure PgBouncer

**Why:** Prevent connection exhaustion under load on the PostgreSQL server.

**How:** Deploy PgBouncer in transaction-mode pooling. Configure `pool_size=20`, `max_client_conn=100`. Set `server_idle_timeout=600`. Connect NocoBase to PgBouncer on port 5432, PgBouncer to Postgres on 5433.

---

### Step 64 — Implement query optimization: EXPLAIN ANALYZE slow queries, add composite indexes

**Why:** Identify and eliminate slow database queries that degrade response times.

**How:** Run `EXPLAIN (ANALYZE, BUFFERS) SELECT ...` on all queries taking >100ms. Add composite indexes for common join patterns: `(entity_id, status)`, `(source_id, credibility_tier)`. Use `pg_stat_statements` to identify hot queries.

---

### Step 65 — Add background job queue: BullMQ for async tasks (email, webhooks, image processing)

**Why:** Offload expensive operations from the request lifecycle to prevent timeouts.

**How:** Install BullMQ + Redis. Define queues: `media-process`, `webhooks`, `notifications`, `credibility-recalc`. Add dead letter queue for failed jobs. Expose job status at `GET /api/jobs/:id/status`.

---

### Step 66 — Implement API rate limiting: per-user, per-IP limits on all endpoints

**Why:** Protect the API from abuse, scraping, and DoS attacks.

**How:** Use `express-rate-limit` or Nginx `limit_req_zone`. Apply: 100 req/min per IP for public endpoints, 1000 req/min per authenticated user. Return `Retry-After` header on 429 responses.

---

### Step 67 — Add CDN integration: serve static assets via Cloudflare or CloudFront with Gzip/Brotli

**Why:** Deliver JS, CSS, images, and media globally with low latency.

**How:** Configure Cloudflare proxy on domain. Enable Gzip + Brotli compression. Set `Cache-Control: public, max-age=31536000` on versioned assets. Purge CDN cache via API on each deploy.

---

### Step 68 — Implement horizontal scaling: multiple app instances behind Nginx load balancer

**Why:** Scale the app to handle higher concurrent traffic without downtime.

**How:** Run 2–4 NocoBase instances. Configure Nginx `upstream dir_app` block with `least_conn` balancing. Use sticky sessions for WebSocket. Share sessions via Redis. Share file storage via mounted volume or S3.

---

### Step 69 — Add health check endpoint: GET /health with DB, cache, memory status

**Why:** Enable orchestrators (Docker, Kubernetes, Render) to monitor app readiness.

**How:** Implement `GET /health` returning `{ status: ok, db: ok, cache: ok, uptime_seconds: N, memory_mb: N }`. Return HTTP 200 on healthy, 503 on degraded. Used by Nginx upstream health checks and cloud load balancers.

---

### Step 70 — Set up monitoring: Prometheus metrics and Grafana dashboard

**Why:** Gain real-time observability into system performance and reliability.

**How:** Expose `/metrics` in Prometheus text format. Track: `http_request_duration_ms` histogram, `http_requests_total` by method+route+status, `db_query_duration_ms`, `cache_hit_ratio`. Import Grafana Node.js dashboard ID 11159.

## Phase 8 — Security Hardening (Steps 71–80)

### Step 71 — Implement input sanitization: strip XSS payloads from all user-submitted fields

**Why:** Prevent stored XSS attacks that could compromise readers and editors.

**How:** Use `DOMPurify` server-side with JSDOM. Sanitize all `TEXT` and `body` fields on write. Allow only safe HTML tags: `p, b, i, a, ul, ol, li, blockquote, code, pre`. Reject `<script>`, `<iframe>`, event attributes.

---

### Step 72 — Add CSRF protection: tokens for all state-mutating requests

**Why:** Prevent forged cross-site requests from exploiting authenticated user sessions.

**How:** Use `csurf` middleware or custom double-submit cookie pattern. Generate per-session token, embed in forms and SPA headers as `X-CSRF-Token`. Validate on all `POST`, `PUT`, `PATCH`, `DELETE` endpoints.

---

### Step 73 — Implement Content Security Policy headers

**Why:** Restrict what resources the browser can load, reducing attack surface.

**How:** Set `Content-Security-Policy: default-src 'self'; script-src 'self'; img-src 'self' data: blob:; style-src 'self' 'unsafe-inline'; connect-src 'self'`. Add `Strict-Transport-Security: max-age=31536000`. Configure via Nginx or app middleware.

---

### Step 74 — Add SQL injection prevention: parameterized queries throughout

**Why:** Ensure no user input can be interpreted as SQL commands.

**How:** NocoBase uses Sequelize ORM which parameterizes by default. Audit all raw queries in migrations and custom actions. Replace any `db.query('...${userInput}...')` with `db.query('...?', [userInput])`.

---

### Step 75 — Implement OAuth2 authentication: Google and GitHub login, JWT session management

**Why:** Provide secure, password-free login and API access.

**How:** Use `passport-google-oauth20` and `passport-github2`. Issue signed JWT access tokens (15 min expiry) + refresh tokens (30 days) stored in `httpOnly` cookies. Implement token refresh: `POST /auth/refresh`.

---

### Step 76 — Add password hashing: bcrypt with salt rounds, enforce strong passwords

**Why:** Protect user credentials if the database is ever compromised.

**How:** Use `bcrypt` with 12 salt rounds. Enforce: min 12 chars, 1 uppercase, 1 number, 1 symbol. Use `zxcvbn` for strength estimation. Never log or transmit raw passwords.

---

### Step 77 — Implement API key system: generate keys for external integrations, revocation support

**Why:** Allow external systems to authenticate with dir without OAuth flows.

**How:** Create `dir_api_keys` table: `(id, user_id, key_hash, name, scopes[], last_used_at, revoked_at)`. Generate 32-byte random keys prefixed with `dir_`. Hash with SHA-256 for storage. Authenticate via `Authorization: Bearer dir_...` header.

---

### Step 78 — Add audit logging: all CRUD operations logged with user, timestamp, diff

**Why:** Provide a tamper-evident trail of every content change for governance and compliance.

**How:** Create `dir_audit_log` table: `(id, user_id, action, table_name, record_id, old_values JSONB, new_values JSONB, ip_address, created_at)`. Hook into NocoBase lifecycle hooks: `afterCreate`, `afterUpdate`, `afterDestroy`.

---

### Step 79 — Implement data encryption: AES-256-GCM for sensitive fields at rest

**Why:** Protect API keys and sensitive metadata even if storage is breached.

**How:** Use `node:crypto` AES-256-GCM. Encrypt `dir_api_keys.key_hash`, optional `dir_users.email`. Store IV alongside ciphertext. Manage encryption key via environment variable or AWS KMS / Google Cloud KMS.

---

### Step 80 — Add security headers: X-Frame-Options, HSTS, Referrer-Policy, X-Content-Type-Options

**Why:** Apply defense-in-depth browser-level security protections.

**How:** Configure in Nginx: `add_header X-Frame-Options DENY; add_header X-Content-Type-Options nosniff; add_header Referrer-Policy strict-origin-when-cross-origin; add_header Permissions-Policy interest-cohort=()`. Run Mozilla Observatory to verify grade A+.

## Phase 9 — Data Export & Integration (Steps 81–90)

### Step 81 — Implement JSON-LD export: export entities as structured linked data

**Why:** Allow semantic web consumers and search engines to parse dir content.

**How:** Expose `GET /api/entities/:slug?format=jsonld`. Output schema.org-compatible JSON-LD: `@type: Thing/Person/Place/Event`. Include facts as `additionalProperty`, sources as `citation`, relations as `sameAs`/`subjectOf`. Validate with Google's Rich Results Test.

---

### Step 82 — Implement CSV export: bulk entity list with metadata columns

**Why:** Allow data analysts and administrators to export content to spreadsheets.

**How:** Expose `GET /api/export/entities.csv?type=person&status=published`. Stream response using Node.js pipeline to avoid memory pressure. Columns: id, slug, title, entity_type, status, credibility_score, fact_count, citation_count, updated_at.

---

### Step 83 — Add RDF/OWL export: generate ontology triples for entities and facts

**Why:** Enable machine-readable knowledge graph export for academic and research use.

**How:** Use `n3` library to serialize as Turtle or N-Triples. Map entities to `owl:NamedIndividual`, entity_type to `rdf:type`, facts to `owl:DatatypeProperty`. Expose `GET /api/export/ontology.ttl`. Register namespace prefix `dir:`.

---

### Step 84 — Implement Wikipedia XML dump import pipeline

**Why:** Bootstrap dir with millions of entities from Wikipedia's open dataset.

**How:** Parse Wikipedia XML dump (`pages-articles.xml`) using streaming SAX parser. Extract: title, intro paragraph, infobox facts, categories. Map to dir schema. Queue entities for board review before publish. Run as BullMQ batch job.

---

### Step 85 — Add Wikidata import: query SPARQL API for structured facts by entity type

**Why:** Enrich entities with verified, structured facts from the world's largest open knowledge base.

**How:** Query Wikidata SPARQL endpoint: `SELECT ?entity ?prop ?value WHERE { ?entity wdt:P31 wd:Q5 }` (humans). Map Wikidata properties (P569=birthdate, P19=birthplace) to dir `fact_type`. Link via `wikidata_qid` field on `dir_entities`.

---

### Step 86 — Create webhook system: POST to external URLs on entity create/edit/publish events

**Why:** Enable external systems (CMS, Slack, CI pipelines) to react to content changes.

**How:** Create `dir_webhooks` table: `(id, url, events[], secret, active, last_triggered_at)`. Dispatch via BullMQ with HMAC-SHA256 signature in `X-Dir-Signature` header. Retry 3x with exponential backoff. Log all delivery attempts.

---

### Step 87 — Implement REST API v2: versioned, paginated, cursor-based

**Why:** Provide a stable, developer-friendly API for external consumers.

**How:** Mount at `/api/v2/`. Add `?cursor=`, `?limit=`, `?filter[entity_type]=person`, `?sort=-credibility_score`. Return `{ data: [], meta: { cursor, total, hasNextPage } }`. Maintain v1 for 6 months post v2 launch.

---

### Step 88 — Add GraphQL API: schema, queries, mutations, subscriptions for real-time updates

**Why:** Provide a flexible query interface for front-end and partner integrations.

**How:** Use Apollo Server or Mercurius. Types: `Entity`, `Fact`, `Source`, `Citation`. Queries: `entity(slug)`, `searchEntities(q, filters)`. Mutations: `publishEntity`, `addFact`. Subscriptions: `onEntityUpdated` via Redis pub/sub.

---

### Step 89 — Implement OAI-PMH: Open Archives Initiative protocol for metadata harvesting

**How:** Implement OAI-PMH 2.0 verbs: `Identify`, `ListMetadataFormats`, `ListSets`, `ListRecords`, `GetRecord`. Format: `oai_dc` (Dublin Core). Expose at `/oai`. Use resumption tokens for large record sets.

## Phase 10 — Production & Operations (Steps 91–100)

### Step 91 — Set up automated backups: daily pg_dump to encrypted S3/GCS with retention policy

**Why:** Protect against data loss from hardware failure, bugs, or ransomware.

**How:** Cron at 02:00 UTC: `pg_dump | gzip | gpg --symmetric --cipher-algo AES256 | aws s3 cp - s3://dir-backups/$(date +%Y/%m/%d).sql.gz.gpg`. Retention: 7 daily, 4 weekly, 12 monthly. Alert on failure via PagerDuty or email.

---

### Step 92 — Implement backup verification: weekly restore test with checksum validation

**Why:** Ensure backups are actually restorable — untested backups are not backups.

**How:** Weekly cron: spin up ephemeral Postgres container, restore latest backup, run `SELECT COUNT(*) FROM dir_entities` and checksum comparison. Send Slack notification with row counts and pass/fail status.

---

### Step 93 — Document disaster recovery plan: RTO and RPO targets, runbook

**Why:** Define how fast dir can recover from a full outage and how much data can be lost.

**How:** Write `docs/disaster-recovery.md`. Targets: RTO ≤ 4 hours, RPO ≤ 24 hours (daily backups). Runbook: DNS failover steps, restore commands, smoke test checklist, escalation contacts.

---

### Step 94 — Set up log aggregation: ship app + Nginx logs to ELK stack or CloudWatch

**Why:** Centralize logs for debugging, security auditing, and operational analysis.

**How:** Use Filebeat or Fluentd to ship logs. Index in Elasticsearch with fields: timestamp, level, request_id, user_id, route, status_code, duration_ms. Set up Kibana dashboards for error rate and traffic patterns.

---

### Step 95 — Implement error tracking: Sentry integration with user context and breadcrumbs

**Why:** Catch production exceptions before users report them.

**How:** Install `@sentry/node` and `@sentry/react`. Set `dsn`, `environment`, `release`. Add `requestHandler` and `errorHandler` middleware. Attach `user.id`, `entity.slug`, and recent DB queries as breadcrumbs.

---

### Step 96 — Add uptime monitoring: Healthchecks.io or Pingdom with SMS/email alerts

**Why:** Know within minutes if dir goes down, before users do.

**How:** Configure HTTP check every 1 min on `/health` endpoint. Alert via email + SMS on 2 consecutive failures. Track monthly uptime SLA. Integrate public status page at `status.dir.yourdomain.com`.

---

### Step 97 — Create CI/CD pipeline: GitHub Actions auto-deploy to Render/GCloud on main merge

**Why:** Eliminate manual deploys and ensure every merge to main is automatically shipped.

**How:** `.github/workflows/deploy.yml`: on push to `main` → run tests → build Docker image → push to GHCR → trigger Render deploy hook or `gcloud run deploy`. Gate on migration check passing. Notify Slack on success/failure.

---

### Step 98 — Implement feature flags: toggle features without redeploy, per-user overrides

**Why:** Ship code to production safely with controlled rollouts and kill switches.

**How:** Use Unleash (self-hosted) or Flagsmith. Store flags in `dir_feature_flags` table: `(name, enabled, rollout_percentage, user_overrides JSONB)`. Check flags in middleware: `if (flag.isEnabled('graphql-api', userId)) ...`.

---

### Step 99 — Add A/B testing: randomized user assignment, metrics tracking, statistical analysis

**Why:** Make data-driven decisions about UI, features, and content presentation.

**How:** Assign users to variant A/B via hashed user ID modulo 2. Track variant in `dir_ab_experiments` table. Log impressions and conversions. Run chi-squared test in analysis scripts to determine winner at p<0.05.
